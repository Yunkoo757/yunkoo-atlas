# Atlas Sync API

Yunkoo Atlas 的个人自托管同步服务。运行时只包含 Node 24、SQLite WAL 与 Caddy，适合 2 GB 轻量服务器。

## 部署

1. 将本目录复制到服务器，例如 `/opt/yunkoo-atlas-sync`。
2. 复制 `.env.example` 为 `.env`。
3. 把 `ATLAS_SYNC_TOKEN` 替换为至少 32 字节的随机令牌。
4. 确保域名解析到服务器，并在云平台实例防火墙开放入站 TCP 80/443；只开放系统内的防火墙不足以完成证书签发。
5. 执行 `docker compose up -d --build`。
6. 访问 `https://<域名>/health`，应返回 `{"status":"ok"}`。

服务不会公开 8787 端口；只有 Caddy 可以访问 API。业务元数据位于 `./data/atlas-sync.db`，原图附件位于 `./data/atlas-sync.db.assets/`，Caddy 证书位于 Docker 命名卷。

## 客户端心流

- 第一台设备：保存服务器地址、资料库 ID 与令牌，然后选择“创建并首次同步”。
- 新建资料库在 7 组核心工作区检查点与原图同步全部完成前保持“未就绪”，其他设备不会拉到半套配置。
- 新设备：输入同一资料库 ID 与令牌，然后选择“连接已有资料库”。该路径不会上传本机空库。
- 所有编辑先落本地库；断网不阻止使用，联网后 outbox 自动补传。
- 原图不压缩、不转码；新设备会按 SHA-256 校验并下载全部引用附件，完成后可离线查看。
- 恢复旧备份后：客户端先暂停自动同步；用户明确选择“以本机重建云端”后，服务端开启新 epoch 并等待完整检查点与附件上传完成。其他设备重新连接时采用新 epoch 权威快照，不与旧历史混合。

## 运维边界

- 修改协议前必须运行 `pnpm test:sync-server` 与项目全量 `pnpm test`。
- 升级服务前同时备份 `data/atlas-sync.db*` 与 `data/atlas-sync.db.assets/`；SQLite WAL 模式下应使用 SQLite 在线备份或先停止 API 容器后复制。
- 令牌只放在服务器 `.env` 与客户端系统安全存储中，不得提交 Git。
- 当前附件单文件上限 32 MB，上传采用原子落盘；同一附件 ID 的不同内容会返回冲突，不会静默覆盖。
- 附件状态会核对服务端文件是否仍存在且大小一致；磁盘文件意外丢失时会按“云端缺失”返回，让仍持有原图的设备自动无损补传。
- HTTPS、Bearer 鉴权和私有服务器不等同于端到端加密；元数据与附件的端到端加密仍属于后续阶段。
