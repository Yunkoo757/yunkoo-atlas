# Yunkoo Atlas 核心产品优化总计划

日期：2026-07-16
状态：已完成（待发布）
分支：`codex/core-product-optimization`
目标：在不引入云同步、多人协作、券商连接或 AI 决策的前提下，把 Atlas 的本地交易记录、复盘、案例学习、数据安全和高频工作台体验收成一个可信、完整、可持续使用的个人交易复盘产品。

## 1. 产品前提

1. Atlas 的核心用户是重视复盘质量、数据主权和长期方法沉淀的个人交易者，而不是需要实时执行或团队协作的机构工作流。
2. 当前最值得解决的不是继续增加宽泛模块，而是打通已有的闭环：`记录 → 执行 → 结算 → 复盘 → 案例 → 再学习 → 统计反馈`。
3. 本阶段以本地优先为产品边界；任何数据能力必须在浏览器与 Electron 的现有能力边界内可解释、可备份、可恢复。
4. 专业感优先来自真实数据口径、可靠恢复、明确状态、稳定交互和统一组件语言，不依赖装饰性视觉。
5. 已批准规格是重要输入，但不能推翻数据不变量；经双重独立产品审查，任何会重复制造业绩事实的复制语义必须修订。
6. 一个真实成交事件只能进入统计一次；知识内容的复制、案例提炼或笔记组织不能自动创造新的成交事实。

这些前提来自用户已确认的产品审计、按推荐顺序实施及“继续直到完成”的指令，不扩展到新的商业模式或外部服务。

## 2. 成功标准

- 仪表盘、策略统计和详情统计使用同一真实口径，不用 `0` 伪装缺失数据。
- 用户完成复盘前必须留下有意义的复盘证据；策略模板能实际进入详情工作流。
- 用户能从历史交易和案例中在 10 秒内开始一轮轻量复盘，并在打开详情后继续原会话。
- 用户从历史记录创建副本时得到新的交易计划或案例，不会复制历史 PnL/R、评论、活动与删除状态。
- 浏览器用户可以重新导入产品推荐导出的 `.journal.zip`，并在提交前看到校验与影响说明。
- 筛选、保存视图、模拟工作区和案例工作区的可见控件与底层支持能力一致。
- 新资料库使用中性、可理解的首次配置；现有资料库不被静默覆盖。
- 关键弹层、文字层级、评论输入、命令搜索、设置页和响应式触摸目标共享清晰的设计语言。
- 全量类型检查、单元测试、浏览器回归、Electron 回归、生产构建、设计契约与固定视口 QA 全部通过。

### P0-A 记录复制语义与统计边界（已完成）

- 移除列表当前“点击复制立即克隆整条对象”的含混行为。
- 实盘与模拟记录的普通复制统一定义为“复制为新计划”：生成新 ID 与 `TRD-*` 编号，重置为 `planned`，清空退出、结算、PnL/R、复盘、评论、活动、删除与案例字段。
- 案例复制生成新 ID 与 `CAS-*` 编号，保留知识正文和直接来源，重置掌握、复看、评论、活动与删除字段。

- “导入历史成交”保留为独立的数据导入语义，不能由普通复制冒充。

### P0-B 数据信任：浏览器完整归档恢复（已完成）

- 浏览器端提供“恢复完整交易库”入口，首版承诺恢复浏览器自身导出的 `data.json + assets/*` 归档。

- 在同一 IndexedDB 事务内替换快照、清除旧附件并写入新附件；任何校验或提交失败都保持旧库完整。
- 错误区分：不是 ZIP、格式不支持、路径非法、版本不兼容、快照非法、附件缺失或重复、容量超限、提交失败。
- Electron 归档与浏览器归档的跨平台互读不在本切片承诺内，遇到桌面格式必须明确提示而非误判。

### P1-A 学习闭环：轻量随机复盘会话（已完成）

- `sessionStorage` 保存同标签页的队列、位置、筛选与翻转状态，不进入持久化 schema。
- 复用案例工作台 scope；“仅含笔记”同时认可正文和纯图片笔记。

验收来源：`docs/superpowers/specs/2026-07-12-random-review-session-design.md`。

## 5. 明确不在本阶段范围

- 账户、云同步、跨设备同步、多人协作、分享与权限。
- 券商连接、自动成交同步、实时行情或交易执行。
- AI 自动分类、AI 交易建议或自动生成策略。
- Anki 式间隔重复算法；随机复盘 V1 只提供临时会话。
- 随机复盘的第二套沉浸阅读模式与长期学习结果持久化；先验证轻量会话是否被持续使用。
- schema v7、复盘笔记本、系统结构化剪贴板及跨工作区自由复制；需求被保留，但在使用证据与数据底座稳定前延期。
- 多账户、多币种换算、券商账户模型；当前版本只声明单一 USD 报告币种。
- 主题市场或完整皮肤系统；本阶段只收口现有暗色系统。
- 自动提交、自动推送、发布或部署；代码保持未提交，除非用户另行授权。

## 7. 关键架构约束

- 业务规则优先写为 `src/lib` 纯函数，React 组件只负责编排和呈现。
- 所有持久化变更通过现有 store 与 storage adapter，不建立第二套浏览器数据库。
- 图片正文始终存 `journal-asset://`，展示时复用 `resolveNoteForDisplay`；资产清理由全局引用决定。
- 新路由复用 `tradeDetailPath` 与详情返回状态；不复制详情页或 Lightbox。
- 随机会话状态不进入主 Zustand store。
- 归档恢复与批量复制必须先完整准备，再原子提交；禁止半成功。
- 新资料库默认变化只作用于 bootstrap 的空库分支。

## 8. 错误与恢复原则

| 流程 | 主要错误 | 恢复方式 | 用户看到 |
|------|----------|----------|----------|
| 随机会话建池 | 池为空、记录会话中被删 | 放宽筛选、跳过缺失 ID | 可操作空态或继续下一条 |
| 笔记解析/图片 | HTML 异常、资产缺失 | 安全文本/图片占位 | 单条降级，不中断会话 |
| ZIP 导入 | 格式/版本/快照/资产错误 | 提交前阻断，不改数据 | 具体错误与重新选择 |
| 批量复制 | 源记录失效、编号冲突、结果字段泄漏 | 纯函数重建目标、一次提交 | 成功数量或明确失败 |
| 自动保存 | 资产保存失败、切换记录 | 保留脏内容并提示重试 | 明确未保存状态 |

## 12. 产品负责人双视角审查记录

### 12.1 前提、杠杆与理想状态

- 核心前提：Atlas 面向一类重视隐私、数据主权和方法沉淀的个人交易者；新资料库需要中性产品体验，已有个人资料不迁移。
- 最高杠杆：可信统计、可验证恢复、极低摩擦的复盘习惯，以及从真实成交到案例再到未来计划的可追溯关系。
- 理想状态：用户可以回答“这笔业绩来自哪次真实成交、如何复盘、提炼成什么案例、下一次如何执行”，且任何知识复制都不会改变历史业绩。

```text
真实成交 ──▶ 结算事实 ──▶ 唯一统计样本
   │              │
   └──▶ 复盘证据 ─┴──▶ 案例/知识引用 ──▶ 新交易计划
                                            │
                                            └── 未真实结算前不进入统计
```

### 12.2 替代路径

| 路径 | 优点 | 主要代价 | 决策 |
|------|------|----------|------|
| 可信账本优先 | 最先建立长期信任，降低数据污染与丢失焦虑 | 短期演示感较弱 | 采用 |
| 学习系统优先 | 差异化强，知识组织以引用为主 | 数据底座未稳时会放大复杂度 | 吸收“引用而非复制”原则 |
| 录入效率优先 | 更接近主流交易日志需求 | 需要券商/账户方向，偏离本阶段边界 | 延期 |
| 个人定制工作台 | 最贴近单一用户当前流程 | 无法形成中性产品体验 | 不采用 |

### 12.3 双视角共识

| 决策点 | 独立审查 A | 独立审查 B | 最终决定 |
|--------|------------|------------|----------|
| 案例/笔记本粘贴到实盘是否保留历史 PnL/R | 反对 | 反对 | 改为创建 `planned` 新计划，清空结果字段 |
| 浏览器完整恢复优先级 | P0 | P0 | 提升到随机复盘之前 |
| 随机复盘范围 | 先轻量单模式 | 先轻量单模式 | 本阶段交付单模式完整闭环 |
| 笔记本与系统剪贴板 | 验证后再做 | 验证后再做 | 本阶段明确延期，无占位入口 |
| `subscribedIds` | 保留兼容、不产品化 | 保留兼容、延期清理 | 不迁移、不暴露含混功能 |
| 竞争策略 | 本地可信复盘 | 本地可信复盘 | 不与券商连接、图表广度和 AI 竞争 |

### 12.4 时序追问与六个月风险

- 如果先做笔记本和自由复制，六个月后最可能发现复盘频率没有提升，而业绩已被副本污染。
- 如果先做随机视觉模式但不做恢复，本地优先会变成数据丢失焦虑，而不是信任优势。
- 如果先做账户/券商扩张，Atlas 会进入成熟竞品最强的横向战场，并丢失当前产品的清晰边界。
- 因此采用选择性扩张：只扩张统计不变量、原子恢复和轻量复盘；其余需求按证据延期。

### 12.5 决策轨迹

1. 原计划把随机复盘放在归档恢复与复制安全之前。
2. 仓库审查确认当前复制会继承历史 PnL/R、评论、活动和删除状态，浏览器完整导出也缺少对应恢复入口。
3. 两位独立产品审查者均认为“二次确认后让历史 PnL/R 进入实盘统计”仍违反产品事实。
4. 计划因此从“功能完整性优先”改为“可信账本与恢复优先”，并把笔记本、schema v7、系统剪贴板移出本阶段完成定义。

## 13. 设计双视角审查记录

现有完整设计审计与独立模型复核达成以下共识：

### 13.1 优先级

- P0：安全复制和完整恢复必须使用清晰的影响预览与原子结果；高风险弹层先共享最小 `ModalShell` 行为。
- P1：随机复盘保持单一连续任务；筛选、URL、保存视图与空状态同义；空库、新建交易、命令搜索和评论输入围绕唯一主动作收口。
- P2：只统一核心文字角色、间距、圆角、断点和触摸目标，不机械迁移全仓组件。

### 13.2 关键流程结构

| 功能 | 结构 | 必须覆盖的状态 |
|------|------|----------------|
| 安全复制 | 触发 → 目标语义 → 保留/清空摘要 → 执行 | 无可复制项、准备中、冲突、全量成功、全量失败 |
| 完整恢复 | 选择归档 → 校验 → 影响预览 → 二次确认 → 原子替换 | 校验中、格式/版本/资产/容量错误、提交中、成功、回滚 |
| 轻量复盘 | 起始筛选 → 正面 → 翻转 → 详情 → 返回原位 → 总结 | 建池中、空池、记录缺失、图片局部失败、结束 |
| 筛选 | 工作区控件 → chips → 保存/清除 → 结果 | URL 恢复、零结果、无效视图、成功保存、未知条件 |
| 空库 | 价值说明 → 新建 → 导入 → 配置策略 | 初始化中、真实空库、导入失败、已有配置但无交易 |
| CommandPalette | 搜索 → 分组结果 → 当前项 → 快捷键 | 无结果、不可用命令、执行、关闭后焦点恢复 |
| TradeComposer | 交易身份 → 参数 → 归档 → footer | 首错聚焦、重复提交锁、无策略、草稿/附件局部失败 |
| 评论 | 安静单行 → 聚焦展开 → 发送 → 列表 | 空、发送中、失败保稿、成功追加、局部附件失败 |

### 13.3 视觉收缩决定

- 不做卡片马赛克、渐变、强阴影、游戏化复盘或第二套沉浸模式。
- `ModalShell` 只统一 overlay、panel、header/body/footer、滚动、Tab 循环、Escape 与焦点恢复，不强制统一业务内容。
- `TradeComposer` 依靠顺序、标题和间距建立层级，不增加装饰分组卡。
- 命令面板只保留低对比外轮廓和 active row；评论框空闲时取消常驻重边界。
- 固定视口验收覆盖 desktop、narrow、compact，并逐项记录 loading、empty、error、success、partial。

### 13.4 最终设计门禁

前期子审查的容量限制已由最终固定视口、交互与差异复核补齐。移动仪表盘、安全复制选择/预览、随机复盘和数据设置均留下视觉证据；workbench 30/30、Linear 26/26、设计契约与 301 图标画廊全部通过。最终产品复核确认缺失样本、策略范围侧栏计数和宽屏触摸网格三项问题已闭环，未发现剩余 P0/P1 设计阻断。

## 14. 工程双视角审查记录

仓库差异审查与独立只读模型复核达成以下结论：

### 14.1 关键裁决

- 安全复制不需要新增 schema 字段；普通副本用现有 `planned`、结果字段和 `sourceTradeId` 即可表达。
- Web 完整恢复必须是独立 `replaceArchive` 流程，禁止复用 JSON 的合并导入。
- ZIP 解压、路径/格式/版本/容量校验和 Blob 准备全部发生在事务外；最终仅用一个 `snapshot + assets` readwrite transaction 执行 `snapshot.put → assets.clear → assets.put(all)`。
- 事务成功后才清空对象 URL 缓存、替换 Zustand/快捷键状态并恢复持久化；失败依赖 IndexedDB 自动回滚。
- 随机复盘快捷键只在路由页捕获阶段生效，排除输入、contenteditable、IME、修饰键、repeat、灯箱和弹层；不写入可配置快捷键 schema。
- `sessionStorage` 只保存版本、资料库标识、ID 队列、cursor、筛选和 flipped；损坏、旧版、越界、缺失记录均清空或降级。
- 完整 `ModalShell` 迁移不阻塞 P0 数据逻辑；最小壳在两个 P0 确认 UI 接入前完成。

### 14.2 最终依赖顺序

```text
1. 安全复制纯转换 + 统计不变量测试
2. 单次 Store 提交 + 复制 UI

4. IndexedDB replaceArchive 原子事务
5. 最小 ModalShell + 两个 P0 确认流程
6. 详情返回契约 + 随机会话核心 + sessionStorage + 键盘
7. 统一 facet + 控件/chip/保存视图/QuickViewBar
8. 新资料库默认值与空库入口
9. 其余高频视觉/可访问性迁移
10. 文档、兼容说明与全量 QA
```
