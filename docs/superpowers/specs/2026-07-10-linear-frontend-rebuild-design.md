# Linear 风格前端重构设计规格

日期：2026-07-10  
状态：已确认

## 1. 背景

当前项目已经具备交易记录、今日快速记录、案例记录、判例、仪表盘、设置、图片存储、导入导出和 Electron 桌面端能力，但前端仍然以页面为单位独立设计：交易工作台、详情、仪表盘和设置分别使用不同的面板、卡片、间距与交互结构。

`docs/linear-frontend-design-system-analysis.md` 已经提取 Linear 的布局、密度、色彩、字体、边框、圆角与动效规律。本次重构不再继续局部换肤，而是将这些规律落实为唯一的应用壳层、页面结构和组件系统。

## 2. 已确认的产品决策

1. 默认交易体验采用 Linear 的列表优先模型，而不是常驻三栏工作台。
2. 点击交易后进入独立详情页；中间展示截图和复盘内容，右侧展示交易属性。
3. 取消侧栏顶部“复盘 / 判例”分段切换器。
4. 判例功能保留，作为侧栏中的独立一级栏目。
5. 时间与策略不再常驻侧栏，迁移到交易列表筛选器，并支持保存常用视图。
6. 普通交易、今日记录和案例记录共用同一种交易行与详情结构。
7. 案例记录继续复用 `Trade` 数据模型，通过 `tradeKind === 'case'` 和复盘分类表达，不进入账户统计。

## 3. 目标

- 建立与 Linear 一致的低噪声、紧凑、稳定且层级清晰的桌面端视觉体验。
- 用一套壳层、列表、详情、筛选器和属性栏承载全部交易类页面。
- 让当日快速记录与周末深度复盘成为同一条连续工作流。
- 保持现有图片、笔记、导入导出、统计边界和本地存储行为不变。
- 通过自动化功能测试和固定视口截图验收重构结果。

## 4. 非目标

- 不照搬 Linear 的品牌、文案或问题管理语义。
- 不重写 Zustand 数据层、IndexedDB / SQLite 适配器或图片资产协议。
- 不合并案例记录与判例的数据模型。
- 不增加云同步、AI 分类、协作和新的统计指标。
- 不在本次重构中更换 React、React Router、Tiptap、Recharts 或 Lucide。

## 5. 设计原则

### 5.1 结构优先

视觉一致性来自稳定结构，而不是大量局部样式。页面必须优先复用应用壳层、工具栏、列表行、分组栏、属性行、菜单和空状态。

### 5.2 单一视觉重心

- 列表页的视觉重心是交易行。
- 详情页的视觉重心是主截图和复盘正文。
- 仪表盘的视觉重心是核心趋势，而不是 KPI 卡片矩阵。
- 设置页的视觉重心是当前表单，不是容器边框。

### 5.3 语义色克制

大面积表面保持低彩度中性色。正负结果、方向、警告和重点状态只在图标、数值、状态点或小标签中使用语义色。

### 5.4 密度稳定

列表行、导航项、工具按钮和属性行使用固定高度。悬停、选中、加载和动态标签不得改变布局尺寸。

### 5.5 渐进披露

默认界面只展示完成当前任务所需的信息。筛选、批量操作、次要属性和完整标签通过面板、菜单或详情页展开。

## 6. 信息架构

侧栏一级导航：

- 今日记录
- 交易日志
- 案例记录
- 仪表盘
- 判例

案例记录的可折叠子视图：

- 全部
- 重点
- 错题
- 待复看
- 已掌握

侧栏底部：

- 回收站
- 设置

时间、策略、状态、品种、方向、交易类型和标签进入列表筛选器。用户可以将筛选组合保存为视图；“本月交易”是内置视图之一，但不占用常驻侧栏。

## 7. 应用壳层

### 7.1 桌面结构

- 侧栏固定宽度：`244px`。
- 主内容与窗口边缘间距：`8px`。
- 主内容框圆角：`12px`。
- 主内容框使用单像素边框和轻微双层阴影。
- 顶部工具栏与内容区域之间只使用一条结构分隔线。

### 7.2 侧栏

- 导航项固定高度 `28px`。
- 选中项使用中性抬升表面，不使用高饱和整块填充或左侧彩色竖线。
- 分区标题只在确有层级时出现，避免每组都增加视觉重量。
- 搜索和新建使用图标按钮，保留工具提示和快捷键说明。
- 策略与时间筛选不再形成长列表。

### 7.3 顶部工具栏

- 左侧显示页面标题、必要的上下文或面包屑。
- 右侧显示视图切换、筛选、排序与页面级操作。
- 保存状态仅在有未保存、保存中或错误时显著显示；已保存状态降级为弱提示。

## 8. 交易列表

### 8.1 基础结构

- 每行高度固定为 `44px`。
- 整行可点击，键盘上下键可移动焦点，Enter 打开详情。
- 复选框只在悬停、键盘焦点或批量选择模式下增强显示。
- 行内不使用独立卡片，不为每条记录添加圆角边框。

### 8.2 信息层级

从左到右：

1. 状态图标。
2. 品种代码与方向。
3. 策略或简短标题。
4. 两个以内的关键标签，超出显示 `+N`。
5. 盈亏与 R 倍数。
6. 日期。

品种是主要文本；策略、标签与日期使用次级或弱文本。金额与 R 倍数保持等宽数值对齐。

### 8.3 分组规则

- 交易日志默认按月份分组，组内按交易日期倒序。
- 今日记录不分组，按创建或交易时间倒序。
- 案例记录默认按复盘分类筛选，不改变交易行结构。
- 分组栏紧凑、可折叠，支持组内快速新建。

### 8.4 筛选与保存视图

筛选器支持：时间、策略、状态、品种、方向、类型、普通标签、错误标签和复盘分类。

- 活跃筛选以紧凑筛选条件显示在工具栏下方。
- 筛选面板与详情侧栏互斥。
- “本月交易”使用 `openedAt` 作为列表筛选口径。
- 第一阶段保留内置视图，不新增复杂的用户自定义视图持久化模型；已存在的常用组合通过固定快捷项提供。

## 9. 交易详情

### 9.1 页面结构

- 顶部：返回入口、交易编号、品种、方向、策略和主要操作。
- 中间主列：标题、主截图、复盘正文、附加截图和活动记录。
- 右侧属性栏：固定宽度约 `440px`，展示可编辑属性。

### 9.2 截图

- 第一张有效截图作为主截图。
- 主截图使用 `object-fit: contain`，不得裁剪图表关键信息。
- 图片加载、缺失和解析失败使用不同状态。
- 点击主截图进入现有灯箱；缩略图切换不得改变主内容宽度。
- 继续使用当前资产 URL 解析和存储协议，视觉重构不得重写图片归属逻辑。

### 9.3 复盘正文

- 编辑器使用无卡片的文档流布局。
- 标题、段落、列表、引用、图片和任务列表使用统一排版节奏。
- 自动保存逻辑、资源归一化和离开页面前刷新逻辑保持不变。

### 9.4 属性栏

- 使用扁平属性行，不使用字段卡片矩阵。
- 标签、复盘分类和状态作为独立分组。
- 分组使用标题和间距分隔，仅必要时使用边框。
- 判例关联保留为次级分组，不抢占首屏视觉重心。

## 10. 快速记录与案例复盘

### 10.1 今日记录

- 新建交易默认使用当天日期和实盘类型。
- 首次提交只要求满足现有数据模型的核心字段。
- 保存后立即出现在今日列表，用户可以继续创建下一笔。
- 深度笔记、完整属性和复盘分类在详情页补充。

### 10.2 案例记录

- 案例记录与普通交易共用列表行、详情页、图片、笔记和标签组件。
- “重点、错题、待复看、已掌握”是筛选视图，不是独立页面设计。
- 历史案例允许使用任意交易日期，不影响今日记录与账户统计。
- `tradeKind === 'case'` 继续从仪表盘、账户盈亏和策略统计中排除。

## 11. 其他页面

### 11.1 仪表盘

- 顶部保留类型和时间范围筛选。
- 将四张 KPI 卡片改为同一统计带中的紧凑指标。
- 趋势图、策略表现和 R 分布使用连续页面区段，不使用大圆角卡片堆叠。
- 本月统计继续按 `closedAt ?? openedAt` 口径，并明确仅统计已平仓记录。

### 11.2 设置

- 设置导航与表单处于同一平面。
- 表单区使用分组标题、说明和属性行，不嵌套面板卡片。
- 危险操作保留清晰语义色和确认流程。

### 11.3 判例

- 数据模型和功能保持独立。
- 移除模块级视觉切换器后，通过一级导航进入。
- 列表和详情尽量复用全局壳层、工具栏、行和属性模式，但不强行与 `Trade` 合并。

## 12. 视觉令牌

以 `docs/linear-frontend-design-system-analysis.md` 为唯一视觉依据：

- 字体：Inter；数值在需要时使用 JetBrains Mono。
- 主体字号：`12px` 和 `13px`；紧凑面板标题不得使用展示级字号。
- 字重：以 `450`、`500` 为主，`600` 以上仅用于少量强层级。
- 控件高度：`28px`。
- 列表行高度：`44px`。
- 标签高度：`20px` 或 `24px`。
- 圆角：`3/5/8/10/12/14px` 与全圆角，不允许页面任意新增值。
- 动效时长：`100/150/250/350ms`。
- 焦点环：单像素、清晰可见，不能引起布局偏移。

## 13. 响应式与可访问性

- 以桌面端和 Electron 为主要体验。
- 窄屏下侧栏改为抽屉，属性栏改为可关闭侧面抽屉。
- 列表在宽度不足时依次隐藏次要标签、策略补充信息和金额，保留品种、状态、R 倍数与日期。
- 所有图标按钮必须有 `aria-label` 和工具提示。
- 菜单、筛选器、列表和属性编辑支持键盘导航。
- 尊重 `prefers-reduced-motion`。

## 14. 状态规范

所有核心页面必须覆盖：

- 加载中。
- 空数据。
- 无筛选结果。
- 图片加载失败。
- 保存中、保存失败和已保存。
- 导入中与导入失败。
- 数据不存在或已删除。

空状态使用低噪声图标、短标题和单一主要操作，不使用大面积插画卡片。

## 15. 迁移策略

采用设计系统优先的纵向切片迁移：

1. 建立视觉令牌、基础组件和自动化设计约束。
2. 重构应用壳层、侧栏和顶部工具栏。
3. 完成交易列表、筛选器和独立详情页的完整链路。
4. 迁移今日记录和案例记录。
5. 迁移仪表盘、设置、判例和其他辅助页面。
6. 删除旧三栏工作台和废弃 CSS。

每个阶段都必须保持应用可构建、现有回归测试通过，并产生固定视口截图。旧实现只有在对应新实现通过验收后才能删除。

## 16. 验收标准

### 16.1 结构

- 应用仅存在一套桌面壳层和侧栏。
- 交易、今日记录和案例记录共用列表与详情组件。
- 默认交易路由不再渲染常驻三栏工作台。
- 右侧不会同时展示筛选面板和详情属性栏。

### 16.2 视觉

- 桌面端侧栏、外边距、主框圆角、控件高度和列表行高符合本文规格。
- 页面中不存在卡片套卡片。
- 选中状态、标签和语义色不会抢过品种、截图和正文的视觉层级。
- 列表首屏密度稳定，行高不因标签数量变化。

### 16.3 功能

- 新建、编辑、删除、恢复、筛选、排序和视图切换正常。
- 图片导入后与正确交易绑定，并能在列表预览、详情和灯箱显示。
- 案例记录不进入仪表盘统计。
- 本月列表与本月仪表盘口径符合既有约定。
- 导入导出和 Electron 构建不回归。

### 16.4 验证

- `npm test` 通过。
- `npm run build` 通过。
- 核心 Playwright QA 通过。
- 至少验证 `1440x900`、`1920x1080` 和一个窄屏视口。
- 保存交易列表、交易详情、今日记录、案例记录、仪表盘和设置页基准截图。

## 17. 风险与控制

- **风险：** 重构过程中业务逻辑被顺带改写。  
  **控制：** 组件迁移与数据逻辑修改分离，优先复用现有 Store 和纯函数。

- **风险：** 新旧 CSS 同时生效导致页面不一致。  
  **控制：** 新组件使用明确命名空间，按页面迁移后立即移除对应旧入口，最终执行未使用样式扫描。

- **风险：** 图片显示或归属再次回归。  
  **控制：** 保持现有资产解析接口，补充跨列表、详情和导入流程的图片 QA。

- **风险：** 全量视觉重构造成长期不可用分支。  
  **控制：** 每个阶段形成可运行纵向切片，阶段结束后执行构建、回归和截图验收。
