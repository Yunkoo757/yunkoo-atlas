# 复盘案例记录设计

日期：2026-07-04

## 背景

用户需要在复盘模块下关注两类内容：

1. 精准查看本月交易。
2. 积累自己的错题库、重点案例库。

错题库和重点案例库的本质仍然接近日常交易记录：需要品种、方向、策略、入场、出场、止损、R 倍数、截图、标签和复盘笔记。但它们不一定是真实发生的账户交易，也不应该进入仪表盘统计。

现有系统已经有 `Trade` 数据模型、交易编辑器、详情页、截图、标签、错误标签、复盘状态、模拟交易和仪表盘统计。最小且一致的方案是在现有交易模型中增加一种新的记录类型，而不是新建一套完全独立的案例模型。

## 目标

- 在复盘模块下新增独立栏目：`案例记录`。
- 案例记录字段和交易记录基本一致，支持像交易一样复盘。
- 案例记录不进入仪表盘统计、实盘交易列表和策略表现统计。
- 支持本月交易的精准入口。
- 保持现有交易录入、截图、详情页和导入导出能力尽量复用。

## 非目标

- 不改造现有判例库。判例库仍用于争议点裁决，例如 BOS 是否有效、IDM 是否标准。
- 不把案例记录混入真实交易统计。
- 不做复杂知识库系统、间隔重复算法或 AI 自动分类。
- 不重写交易编辑器和详情页。

## 推荐方案

在 `TradeKind` 中新增第三种类型：

```ts
type TradeKind = 'live' | 'paper' | 'case'
```

语义：

- `live`：实盘交易，默认进入交易列表和仪表盘。
- `paper`：模拟交易，进入模拟列表，可在仪表盘通过类型切换查看。
- `case`：复盘案例记录，进入案例记录栏目，不进入仪表盘统计。

案例记录复用 `Trade` 的主体字段：

- 品种：`symbol`
- 方向：`side`
- 状态：`status`
- 策略：`strategyId`
- 普通标签：`tags`
- 错误标签：`mistakeTags`
- 复盘状态：`reviewStatus`
- 入场、出场、止损、仓位、盈亏、R 倍数
- 开始日期、结束日期
- 富文本笔记和截图
- 评论和活动记录

## 导航与视图

复盘模块侧栏建议结构：

- 工作台
  - 交易
  - 案例记录
  - 仪表盘
- 时间
  - 今日
  - 本周
  - 上周
  - 本月
  - 上月
- 智能视图
  - 进行中
  - 星标交易
  - 错过的机会
  - 模拟

`案例记录` 使用独立路由 `/review-cases`。不使用 `/cases/review`，避免和现有判例库的 `/cases` 路由混淆。

列表标题应明确它不是账户交易，例如：

`案例记录`

副标题：

`不进入仪表盘统计 · 用于沉淀错题、重点案例和经典形态`

## 案例记录筛选

案例记录栏目建议先提供这些筛选：

- 全部
- 重点
- 错题
- 待复看
- 已掌握

为了避免新增过多字段，第一版可以复用现有字段表达：

- 重点：`reviewStatus === 'focus'`
- 错题：`mistakeTags.length > 0` 或 `status === 'missed'`
- 待复看：`reviewStatus === 'unreviewed'`
- 已掌握：`reviewStatus === 'reviewed'`

如果后续需要更细，可以再增加 `caseCategory` 或 `studyStatus` 字段。第一版先不加，减少迁移和表单复杂度。

## 状态文案

案例记录可以继续使用现有 `TradeStatus`，但在案例栏目里显示更贴合学习语境的文案：

- `planned`：观察中
- `open`：推演中
- `missed`：错题
- `win`：正例
- `loss`：反例
- `breakeven`：中性

这只是显示层映射，不改变底层状态含义，避免影响已有交易逻辑。

## 本月交易

交易列表需要补齐时间入口：

- 今日
- 本周
- 上周
- 本月
- 上月

列表时间视图默认按 `openedAt` 筛选，表达“本月发生或计划的交易”。

仪表盘需要增加 `本月` 时间范围，按 `closedAt ?? openedAt` 统计，表达“本月产生结果的交易”。仪表盘标题或副标题继续明确：`仅统计已平仓 · 按平仓日累计`。

## 统计边界

所有交易统计必须显式排除 `tradeKind === 'case'`：

- 仪表盘默认统计实盘，不包含案例。
- 仪表盘选择全部类型时，也只包含 `live` 和 `paper`，不包含 `case`。
- 策略表现统计不包含案例。
- 实盘交易数量不包含案例。
- 活跃交易、星标交易、错过机会默认不包含案例，除非当前页面就是案例记录。

实现时应集中封装过滤函数，例如：

```ts
function isAccountTrade(trade: Trade): boolean {
  return trade.tradeKind === 'live' || trade.tradeKind === 'paper'
}

function isReviewCaseTrade(trade: Trade): boolean {
  return trade.tradeKind === 'case'
}
```

避免各页面重复写 `tradeKind !== 'case'`，降低以后漏统计的风险。

## 创建与转化

第一版支持两种创建方式：

1. 在案例记录栏目点击新建，默认 `tradeKind = 'case'`。
2. 在交易详情或列表菜单中增加“沉淀为案例”，复制当前交易生成一条 `case` 记录。

复制为案例时建议：

- 保留品种、方向、策略、标签、错误标签、价格、R 倍数、日期、笔记和截图。
- 生成新的 `id` 和 `ref`，例如 `CAS-001` 或继续使用现有编号体系但前缀不同。
- 在笔记开头补一行来源，例如：`来源交易：TRD-142`。
- 不自动关联到判例库。

## 数据迁移

现有数据只有 `live` 和 `paper`。迁移规则：

- 缺失 `tradeKind` 的旧数据继续归一为 `paper` 或现有兼容逻辑。
- 新增 `case` 后，`normalizeTradeKind` 必须允许 `case`，不能把它归一成 `paper`。
- 导入导出需要保留 `case` 类型。

## 测试计划

至少补充以下回归测试：

- `normalizeTradeKind('case')` 返回 `case`。
- 仪表盘统计排除案例记录。
- 交易列表默认不显示案例记录。
- 案例记录列表只显示 `tradeKind === 'case'`。
- 本月列表按开仓日筛选。
- 仪表盘本月按平仓日筛选。
- “沉淀为案例”生成新记录，不修改原交易。
- 导入导出保留案例记录。

## 风险与取舍

主要风险是统计口径泄漏：案例记录复用 `Trade` 之后，如果某个统计页面没有排除 `case`，就会污染数据。

应对方式：

- 增加 `isAccountTrade` 和 `isReviewCaseTrade` 这类集中判断函数。
- 让 Dashboard、策略统计、侧栏计数全部走同一套判断。
- 给关键统计写回归测试。

这个方案比新建独立模型更省实现和维护成本，也更符合用户“本质上跟日常记录的交易案例一致”的表达。
